1 THE BET

Chronos bets that strategic leaders -- CTOs, PMs, Founders -- will adopt a personal AI that captures their thinking from wherever it happens, synthesizes it into compounding knowledge, and retrieves it grounded in their own reasoning. The market has capture tools (Granola, Rewind), organization tools with AI bolted on (Notion, Obsidian), and consumer-grade AI notes (Mem) -- but nothing that combines intent detection, role-specific profiles, multi-channel capture, and a knowledge base that compounds over time. This is the whitespace. The first 3 months are a foundation investment: engine, memory architecture, capture loop, sessions, and Discord as the primary channel. If the foundation is right, the mid-horizon delivers the first intent that proves the full pipeline end-to-end (clarify), and the long horizon delivers the differentiators that make the product defensible (proactive synthesis, role profiles, channel expansion). The moat is the user's own knowledge -- every week of use makes switching harder and retrieval more valuable.

3 KEY DECISIONS

The following decisions require explicit reviewer agreement. Each changes sequencing, priority, or scope if reversed.

D1: Foundation-heavy near horizon (5 of 5 near-term epics are foundation investments)

All five near-horizon epics (E1, E2, E3, E4, E8) are tagged as foundation investments. This means the first 3 months deliver infrastructure, not user-facing differentiation. The MVP value loop (capture via Discord, heartbeat classifies, knowledge becomes retrievable) is the minimum viable proof point, but it does not yet include the intent-driven reasoning that differentiates Chronos from simpler capture tools.

Trade-off: Slower time to differentiated value. Faster time to stable architecture that mid-horizon builds on without rework.

If reversed: Pulling E5 (clarify pipeline) or E6 (confidence scoring) into near horizon increases near-term scope by ~2 XL-effort items and introduces risk of building on an unstable memory layer. E4's adapter interface must stabilize first.

D2: E4 (Two-Layer Memory) as the highest-risk near-term investment

E4 carries HIGH RISK. Two blockers affect downstream sequencing: (1) Layer 1 source of truth must be pinned before the adapter interface is defined -- this cascades to E2 and E8. (2) Upstash Vector free tier rate limits must be validated before committing to the search index architecture.

Trade-off: If Layer 1 source or search index choice changes mid-build, E2 (heartbeat), E8 (sessions), and everything downstream shifts.

If reversed: Deferring E4 would force E2 and E8 to build against a mocked memory interface, adding rework when the real adapter arrives. Alternatively, simplifying to a single-layer memory reduces near-term risk but removes the quality-gating differentiator from the architecture.

D3: Discord as sole MVP input channel (not Claude Code, not web)

Discord is the primary user interface for MVP. Claude Code connects as a client channel but is scoped for code/file work locally and engine API calls for context operations. Web is prototype-only in near horizon. The bet is that Discord slash commands provide sufficient capture UX for early validation.

Trade-off: Users who prefer CLI-first interaction wait until E12 (long horizon) for a fully supported Claude Code channel. Discord's 3-second interaction acknowledgment window requires a deferred response pattern from day one in E1.

If reversed: Adding Claude Code as a co-primary channel in near horizon splits integration effort and complicates the "one engine, multiple channels" validation. Discord is simpler to validate because it has a constrained interaction model (slash commands).

D4: E9 (Intent Breadth) scoped as XL but deferred to P2 mid-horizon

E9 bundles five distinct intent implementations (decide, validate, consult, advise, design) into one XL epic at P2 priority. This is the largest scope item on the roadmap after the near-horizon foundation. The feasibility flag notes that five intents hidden under XL needs decomposition, and chained Opus synthesis calls risk hitting the 60-second timeout.

Trade-off: Keeping E9 as one epic simplifies roadmap tracking but obscures the actual work. Decomposing it into 3-5 smaller epics would reveal true sequencing constraints between intents.

If reversed: Promoting E9 to P1 mid-horizon or pulling any individual intent forward changes the mid-horizon scope significantly and may compete with E5 (clarify) for capacity.

4 WHAT WE ARE NOT DOING

Multi-user / team features. MVP is owner-only. No trust tiers, no guest access, no shared memory, no collaboration. The architecture supports it; the roadmap does not include it within 12 months.
Mobile-native clients. Discord on mobile is the mobile story. No native iOS/Android app. No voice capture beyond what Discord provides.
Self-hosted / on-premise deployment. Vercel Hobby plan is the deployment target. No Docker images, no self-hosted option, no enterprise deployment model. Privacy concerns are addressed by data ownership terms and encryption, not by deployment model.
PM and Founder persona profiles in the 12-month horizon. E10 abstracts the role-profile system so future personas can be added via markdown. But only the CTO profile is built. PM and Founder profiles are beyond this roadmap.
Import / migration tooling. No "import your Notion" or "import your Obsidian vault" wizard. Cold start is addressed by guided onboarding and fast capture UX, not by bulk import. This is a deliberate scope cut despite the cold-start risk.
LLM provider portability. The roadmap is Claude-native (Haiku for scoring, Sonnet for routing, Opus for synthesis). No multi-provider abstraction layer. Anthropic pricing and capability trajectory is a monitored risk, not a near-term engineering investment.

5 ASKS

ASK 1: Approve the foundation-heavy near horizon.

The first 3 months are all infrastructure. The earliest user-facing differentiation (intent-driven clarify) is mid-horizon. Confirm this sequencing is acceptable given the product timeline and any external commitments (demos, investor updates, user pilots).

ASK 2: Decide on E4 Layer 1 source of truth before E4 starts.

The memory adapter interface cannot be designed until the Layer 1 storage is pinned. Candidates need to be evaluated (e.g., Obsidian vault via git, Postgres-native, filesystem). This decision cascades to E2, E8, and all downstream epics. It is the single highest-leverage decision on the roadmap and must be resolved before engineering begins on E4.

ASK 3: Confirm E9 decomposition approach.

E9 bundles 5 intents into one XL epic. The feasibility assessment flags this as needing decomposition. Options: (a) decompose now into 3-5 smaller epics with independent sequencing, (b) keep as one epic and decompose at mid-horizon planning, (c) pick 2-3 priority intents and defer the rest. This affects mid-horizon capacity planning.

ASK 4: Validate Upstash Vector free-tier viability for E4.

E4 depends on Upstash Vector for the search index layer. Free-tier rate limits have not been validated against expected query volume. If rate limits are insufficient, the alternative is self-managed vector search, which changes E4 effort and potentially its timeline. This validation should happen before E4 begins.

6 ASSUMPTIONS

A1: Solo builder capacity is sufficient for the near-horizon scope.

The near horizon contains 2 XL epics (E1, E4), 1 M epic (E3), and 2 L epics (E2, E8) -- all P1. This is scoped for a single builder over 3 months. If capacity is lower than expected, E2 (heartbeat) is the most likely deferral candidate because the capture loop works without LTM promotion (it just does not compound yet).

A2: Vercel Hobby plan constraints are acceptable for MVP.

Vercel Hobby plan limits: 1 cron/day (accepted for heartbeat in E2), 60-second function timeout (risk for E5 and E9 multi-agent orchestration), and hobby-tier bandwidth. The roadmap assumes these limits hold for the full 12 months. If usage outgrows Hobby, upgrading to Pro changes the cost model but not the architecture.

A3: Discord's 3-second acknowledgment window is solvable with a deferred response pattern.

Discord Interactions API requires an initial response within 3 seconds. All engine processing beyond trivial lookups exceeds this. The assumption is that a deferred response pattern (acknowledge immediately, send follow-up when processing completes) works reliably enough for MVP. This pattern must be designed and validated within E1 scope, not deferred.

A4: Intent detection accuracy is achievable with current LLMs at useful cost.

The vision targets 85%+ intent detection accuracy across five core intents. The roadmap assumes current Claude models can achieve this without fine-tuning. If accuracy is materially lower, E5 (clarify) and E9 (intent breadth) timelines shift right as more iteration is needed on the detection pipeline.

A5: The CTO persona is sufficient for early validation.

Starting with CTO-only constrains the addressable market but focuses the intent model and knowledge taxonomy. The assumption is that validating with one persona proves the architecture generalizes. If it does not, the role-profile abstraction in E10 carries more risk than estimated.
